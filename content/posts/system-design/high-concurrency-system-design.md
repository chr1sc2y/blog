# 高并发系统设计：从流量削峰到库存一致性的工程取舍

设想一个最朴素的下单接口：用户点击购买，服务收到请求，开启一个数据库事务，`SELECT` 出库存，判断大于零就 `UPDATE` 减一，提交。平时这套逻辑跑得很好。直到某个整点，一款爆品开放一万件库存的秒杀，前一秒还是几百 QPS，下一秒涌进上百万请求——所有请求都在抢同一行库存记录的行锁，数据库 CPU 瞬间打满，连带把正常的交易链路一起拖垮。

这是高并发系统设计绕不开的起点：**当流量的量级跨越几个数量级时，原本正确的逻辑会因为"正确得太慢"而变成故障源**。解决它不是某一个银弹，而是一组分层的取舍——把流量在每一层尽可能地削减、拦截、异步化，让真正落到最脆弱的那一层（数据库）的压力降到可控。下面按照请求从入口到落库的路径，逐层拆解。

## 1 流量削峰：把洪峰摊平在每一层

高并发的第一原则是：**不要让秒杀流量打到为常规业务设计的链路上**，更不能直接打到数据库。流量需要在抵达核心逻辑之前被逐层削减。

```
用户 ──► CDN ──► API 网关 ──► 秒杀服务（无状态）
                              ↘ Redis（库存 & 幂等标记）
                              ↘ MQ（异步下单） ──► 订单服务 ──► DB
```

这条链路上的每一层都承担了一部分"削峰"职责：

- **CDN** 把活动页、商品详情、倒计时脚本这类静态资源提前缓存到边缘节点，绝大部分静态请求根本不会回源。这一层挡掉的往往是流量的大头。
- **API 网关** 作为统一入口，做鉴权、IP/用户维度限流、黑名单和人机校验。恶意刷单和机器人流量应该止步于此。
- **秒杀服务** 本身保持无状态、只做轻逻辑，因而可以近乎无限地水平扩容。把无状态作为硬约束，意味着扩容只是加机器，不需要考虑会话迁移。
- **消息队列** 通过缓冲请求实现"削峰填谷"，把瞬时洪峰摊平成下游能消化的平稳速率，同时解耦生产者与消费者。

值得强调的是限流与熔断的区别，这两个词经常被混用：**限流**是限制单位时间的请求量，防止系统被压垮，是一种主动的流量整形；**熔断**则是在探测到下游已经异常时主动切断请求，防止故障沿调用链扩散。前者防患于未然，后者止损于既发。

## 2 缓存与热点：单点是最先崩的地方

把库存和判重逻辑放进 Redis，是因为内存级的响应速度和原子能力远非数据库可比。但缓存层并非高枕无忧——**即使做了分片，一个极热的 Key 依然会打爆它所在的单一节点**。当某个商品的 QPS 达到十万量级，所有请求经哈希都落到同一个 Redis 实例上，瓶颈从数据库转移到了这个节点的 CPU 和网卡。

热点治理通常有两级防御，它们的取舍很值得展开。

**第一级是热点 Key 复制（Key Replication）**：把热点 Key 复制成 `key_1 … key_N` 多份，由哈希分散到集群的不同物理节点；读取时在副本间随机或轮询选择。

```
        热点 Key                      复制为 N 份
      stock:1001         ──►   stock:1001:1  ──► Node A
   (单节点被打爆)                stock:1001:2  ──► Node B
                                stock:1001:3  ──► Node C
                                    ...
                              读取时随机选一份 ──► 负载被打散
```

它的好处是读性能随副本数线性扩展，且应用层保持无状态、运维简单；代价是写放大——一次更新要写 N 份，带来毫秒级的数据延迟。

**第二级是本地缓存（Local Cache）**：在应用进程内（如 Caffeine）直接缓存最热的数据，设一个很短的 TTL，请求在应用层就返回，根本不走网络去访问 Redis。它的延迟最低、对后端的保护最彻底，但代价是有状态——消耗应用内存、有 OOM 风险，且多实例间会有数秒的不一致。

那么两者怎么选？我的判断是看瓶颈到底在哪：**对于常规热点，优先用 Key Replication**，因为它保持了应用层无状态，能直接吃到 Redis 集群的弹性；**而对于现象级热点**（那种瞬间冲上来、瓶颈已经不在 CPU 而在网络带宽的流量），就必须上 Local Cache，因为只有彻底消除网络 I/O 才扛得住。换句话说，Key Replication 解决的是"算不过来"，Local Cache 解决的是"网络传不过来"，这是两个不同维度的问题。

至于缓存层自身的高可用，一套常见的三层保护是：集群分片加多副本加自动故障转移，保证服务不中断；缓存故障时立即 fail-fast 并熔断，宁可拒绝写请求也不让流量穿透到数据库（一致性优先）；万一缓存数据丢失，则依赖消息队列里已经成功落库的记录回放，用"总量减去已处理量"重建状态。这套设计的内核是：**在故障面前，明确地选择牺牲什么**——这里牺牲的是部分请求的可用性，换取库存绝对不超卖。

## 3 库存扣减：从加锁到无锁的演进

库存扣减是整个系统正确性的核心，因为它直接对应"不超卖"这条铁律。这里的方案演进，本质是一条从"能用"到"极致吞吐"的取舍路径。

| 方案 | 问题所在 |
| --- | --- |
| 数据库悲观锁（行锁） | 数据库成为瓶颈，高并发下被锁竞争拖垮 |
| 乐观锁（CAS / 版本号） | 高冲突场景引发重试风暴，CPU 空转飙升 |
| 分布式锁 | 把并行请求串行化，单热点 Key 限制了吞吐上限 |
| 无锁原子脚本 | 在缓存内部原子执行查减，无锁竞争——**推荐基准** |
| 库存分桶 | 把单 Key 拆成多桶分散到不同节点——**终极优化** |

前三种方案的共同困境是：它们都试图用"锁"来保证正确性，而锁的本质是把并行变成串行，这恰恰与高并发的目标背道而驰。乐观锁看似无锁，但在秒杀这种极高冲突的场景下，绝大多数 CAS 都会失败重试，重试风暴本身就是一场灾难。

真正的转折点在于**把"检查—扣减—判重"这三步合并成 Redis 内部的一段原子脚本**。因为 Redis 单线程整体执行脚本、不会被打断，所以这三步天然原子：库存大于零才扣、同一用户只能成功一次。它既消除了应用层的锁竞争，又把三次网络往返压缩成一次。这是性能与正确性兼顾的基准方案。

而当单个热点 Key 的吞吐仍然见顶时，终极手段是**库存分桶**：把一万库存拆成 `stock:1001:1` 到 `stock:1001:N` 共 N 个逻辑 Key，分散到集群的不同节点，写性能近乎线性提升 N 倍。但分桶不是免费的，它引入两个必须正面解决的衍生问题：

- **库存碎片化**：某个桶卖空了、其他桶还有货。正确的做法不是去做复杂的"库存搬运"，而是**请求二次路由**——脚本发现当前桶为空就返回一个信号，应用层随机尝试下一个桶。把复杂度留在无状态的应用层，而不是污染库存数据本身。
- **缓存与数据库的一致性**：要防止"Redis 扣减成功、但发往 MQ 的消息丢了"这种状态撕裂。事务型消息（先发 half message，再执行扣减，最后按结果提交或回滚）能把这两个动作绑成最终一致。

## 4 异步落库与幂等：让数据库按自己的节奏写

扣减成功之后，订单不应该同步写数据库——那等于把数据库重新拉回了主链路。正确的做法是把"扣减成功"这件事作为一条消息投递到 MQ，由订单服务异步消费、平稳落库。

```
Redis 扣减成功 ──► 投递消息(requestId) ──► MQ
                                          │
                                          ▼
                            订单服务消费 ──► 幂等校验(requestId 唯一)
                                          ──► 批量写入 DB
                                          ──► 失败 ──► 重试 / DLQ
```

这里有两个工程细节决定了系统在异常下是否还能自洽：

第一是**幂等性**。网络重试、消息重投在分布式系统中是常态，因此消费端必须用唯一的 `requestId` 做幂等校验——同一个请求落库多次也只产生一条订单。需要厘清的是，锁只能挡住并发，挡不住重试带来的重复提交，所以幂等是与锁正交的、必须单独保证的一层。

第二是**消费速率的弹性**。当消费跟不上、消息开始堆积时，处理思路有三个层次：提升消费并发（加分区、加消费者）、优化下游写入（批量插入是最有效的）、以及在生产侧主动限流（监控 backlog，超阈值就对新请求返回"活动火爆，请稍后再试"）。最后这一招尤其关键，它承认了一个朴素的事实——**当下游确实消化不了时，优雅地拒绝一部分请求，远好过让整个系统雪崩**。

## 5 当延迟以微秒计：另一个极端

前面讨论的都是"高吞吐"场景，把延迟容忍到秒级、用异步换吞吐。但还存在另一类系统，目标恰恰相反：**在微秒级延迟下处理千万级吞吐**，典型如高频交易的行情处理。把它放在这里对照，是因为它会逼着你重新审视前面所有"想当然"的设计。

在这个量级下，许多常规手段直接失效，取舍也走向另一个极端：

| 维度 | 常规高并发 | 微秒级低延迟 |
| --- | --- | --- |
| 接入协议 | TCP，保证可靠 | UDP 组播，**实时性 > 完整性**，丢包不重传只标记"不可信" |
| 扩容 | 自动扩缩容 | 拒绝自动扩缩容（启动太慢、调度抖动），改用静态分区 + CPU 核绑定 |
| IO | 异步落库 | 主路**零 IO**：内核旁路（DPDK），收包-算策略-发单，严禁写盘打日志 |
| 内存 | 按需分配 | 对象池预分配，用数组下标代替指针，消除运行时 GC/malloc |

这里最反直觉的是"实时性优先于完整性"：在交易场景里，**过时的数据不仅无用，反而有害**——基于一个几毫秒前的旧价格下单可能直接造成亏损，所以一旦检测到丢包，正确的反应不是等重传，而是立即把订单簿标记为不可信、触发熔断、再走快照通道重建。这和秒杀里"宁可拒绝也不超卖"是同一种思维方式：**先想清楚在故障下要保住什么，再据此设计降级路径**。

## 6 强一致与防超卖：日历类系统的另一面

并不是所有高并发场景都能接受最终一致。像住宿预订这类日历系统，最高优先级是强一致——绝不允许重叠预订。它读多写少，但热门房源在高峰也会出现秒杀级的写入，于是它同时要面对"强一致"和"高并发"这对看似矛盾的诉求。

它的几个设计选择很有代表性：

- **数据库选型**：尽管 NoSQL 扩展性更好，但为了严格防止重叠预订，关系型数据库的 ACID 事务与 `SELECT ... FOR UPDATE` 行锁更稳妥；NoSQL 的乐观锁在热点下又会回到重试风暴的老问题。这是一个为了正确性主动放弃部分扩展性的取舍。
- **按天一行**：每个房源每天一行数据，提供最细粒度的行锁——订 5 月 1 日和订 5 月 20 日的两个请求可以并行写入、互不阻塞。数据模型的粒度，直接决定了并发度的上限。
- **两阶段状态机**：把"预占库存"和"支付"拆开。预占阶段开启短事务、锁定日期行、把状态置为 `HELD` 并设过期时间，随即提交释放连接；支付走在事务之外调用第三方；最后用 webhook 回调加定时对账兜底——对账任务扫描过期的 `PENDING` 订单，已付的补齐、未付的释放库存。**把长事务拆成短事务加补偿，是高并发下处理跨系统流程的通用套路。**

而真遇到极热房源（短时间内上万人抢同一间），仍然回到第 2、3 节的老办法：在数据库前加一层热点服务，用 Redis 原子脚本拦掉绝大多数请求，只放行成功者异步落库，数据库的总库存作为最终兜底。可以看到，**不同业务的高并发方案，到最后往往收敛到同一组模式上**。

## 整体而言

回头看这一整条链路，高并发系统设计其实没有什么神秘的银弹，它是一系列围绕同一目标的分层取舍：用 CDN 和网关在外层削掉大头，用缓存和原子脚本把核心逻辑挡在数据库之前，用消息队列把写入异步化、让数据库按自己的节奏工作，再用幂等和对账保证异常下的自洽。

贯穿其中的，是两个反复出现的工程判断。一是**把脆弱的资源（数据库、单个热点节点）保护在层层防线之后**，越往里的层越要"省着用"。二是**在故障来临前就想清楚要牺牲什么**——是牺牲一部分请求的可用性来保住一致性，还是牺牲数据的完整性来保住实时性。能把这两个判断在每一层都讲清楚，一个高并发系统的设计也就立住了。
