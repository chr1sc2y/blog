# 工业级 Agent 开发：模型之外，你真正在打磨的是 Harness

很多人第一次做 Agent 的路径是这样的：调一个能力够强的模型，把任务写进提示词，让它自己调几个工具跑起来。demo 阶段效果惊艳，可一旦放到真实业务里，问题就接连冒出来——工具被乱调、长任务跑到一半断了没法恢复、改了个提示词线上行为悄悄退化却没人发现、出了问题完全不知道是哪一步错的。

这些问题的共同点是：**它们都不在"模型"这一层，而在模型外面那层工程外壳里**。我更愿意用一个简单的等式来概括工业级 Agent 的真实构成：

```
Agent = Model + Harness

Harness =
  instructions        指令
  tool interface      工具接口
  state machine       状态机
  memory / context    记忆与上下文
  workflow graph      工作流图
  permission system   权限系统
  human approval      人工审批
  tracing / logging   追踪与日志
  eval runner         评测
  sandbox             沙箱
  retry / resume      重试与恢复
  deployment runtime  部署运行时
```

模型是买来的、共享的、会不断被更强的版本替换；而 harness 是你自己长期积累、真正构成壁垒的部分。所以构建一个 Agent 的过程，本质上是**逐步打磨 harness 的过程**：让能力边界扩大，让失控空间缩小，让行为结果变得可验证。可靠性几乎从来不是靠一句更长的提示词单独获得的，而是由工具边界、状态机、评测集、审批机制和可观测性共同形成的。

理解了这个等式，下面这条从单 Agent 到生产系统的路线，就有了一根贯穿始终的主线。

## 1 渐进式路线：不要过早上多 Agent

一条务实的路线通常从单 Agent 和函数调用开始，再依次加入检索、状态、审批、评测和部署。一个常见的错误是过早引入多 Agent——在工具、状态和评测这些基础还没稳定之前就上多 Agent，只会让调试难度成倍上升。

| 阶段 | 核心能力 | 工程产物 |
| --- | --- | --- |
| 模型调用 | 提示词、结构化输出、多轮上下文、流式输出 | 能返回 JSON 的单步任务处理器 |
| 工具调用 | 函数 schema、工具错误处理、结果压缩、dry-run | 只读工具、低风险写入工具、调用日志 |
| 工作流 | router、planner-executor、evaluator-optimizer、状态机 | 可暂停和恢复的任务流 |
| 安全边界 | 权限、审批、沙箱、数据边界、prompt injection 防御 | 高风险动作确认机制和审计记录 |
| 评测与观测 | trace、eval、回归测试、成本和延迟统计 | 失败样本库、golden traces、监控面板 |
| 部署运行 | 队列、持久状态、重试、幂等、取消、灰度 | API、worker、数据库、对象存储、日志 |

值得注意的是这张表的推进顺序：能力（前两行）在前，约束（后四行）在后。但约束并不是"做完能力之后的可选项"，它们恰恰是把一个能跑的 demo 变成能托付的系统的关键。

## 2 开源实现的参考层次

工业级 Agent 系统几乎不需要从零实现所有基础设施。更现实的做法是复用成熟的开源框架和观测工具，把自己的精力留给业务工作流、权限策略、评测标准和审批体验这些真正承载业务责任的部分。

| 类别 | 代表项目 | 适用位置 |
| --- | --- | --- |
| Agent 编排 | OpenAI Agents SDK、LangGraph、Microsoft Agent Framework、CrewAI | 工具调用、handoff、工作流图、多 Agent 编排 |
| 应用平台 | Dify | 快速搭建带 UI、RAG、workflow 的 LLM 应用 |
| 代码执行型 Agent | OpenHands | 长任务、沙箱、代码修改、任务回放 |
| 观测与评测 | Langfuse、Arize Phoenix | trace、prompt 管理、评测、线上样本分析 |
| 工具协议 | Model Context Protocol | 统一连接数据库、文件、GitHub、浏览器、内部系统 |

但这里有一条边界需要划清楚：开源框架适合承担通用运行机制，**业务责任边界却必须单独建模**。是否允许发送邮件、修改生产数据、删除文件、调用付费 API——这些决定权不应该交给某个通用框架的默认行为，它们是你自己业务风险模型的一部分。

## 3 从零构建，还是基于开源构建

当目标是一个业务 Agent，最常见也最合理的组合是"基于开源构建，核心业务 harness 自主设计"。哪些该复用、哪些该自己做，其实有一条相当清晰的分界线：

| 更适合复用 | 更适合自主设计 |
| --- | --- |
| Agent SDK、workflow graph、MCP、RAG 基础组件 | 业务 workflow、任务状态、领域工具 schema |
| Langfuse / Phoenix 等观测与评测平台 | 任务成功标准、失败分类、线上回归样本 |
| 容器、队列、对象存储、托管数据库 | 权限升级、审批体验、回滚策略、审计格式 |
| 沙箱、浏览器自动化、代码执行环境的成熟底座 | 进入上下文的数据范围、高风险动作限制 |

可以看出，左列几乎都是"与业务无关的通用机制"，右列几乎都是"与业务风险强相关的判断"。复用左列是为了不重复造轮子，自研右列是因为它们决定了系统在你的具体场景下是否安全、是否可验收——这部分没法外包给框架的默认值。

## 4 最难的部分：评测、权限、状态、可观测

工业级 Agent 的主要难点，几乎都不在"调用模型"本身，而在它周围这几件事上。按我的经验，下面六点是真正决定一个 Agent 系统能否长期运行的地方：

1. **评测体系**。没有 eval，提示词、模型或工具的任何一次变更都可能造成隐性退化而无人察觉。任务集需要覆盖正常、边界、恶意、高风险和长上下文样本。
2. **工具设计与权限边界**。工具接口过宽会放大误用风险，过碎又会拖低效率；高风险工具必须配 dry-run、确认、审计和回滚路径。
3. **状态管理与可恢复性**。长任务会中断、超时、失败、等待审批——checkpoint、resume、幂等和取消机制，直接决定系统能不能跑得久。
4. **Tracing 与可观测性**。生产调试需要知道每一次运行的输入、工具调用、工具结果、模型输出、成本、延迟和失败位置；这些必须从设计之初就内建。
5. **Context engineering**。上下文需要被检索、压缩、排序和丢弃。一个反直觉但重要的事实是：上下文越长，并不等于结果越可靠。
6. **Human-in-the-loop**。工业系统往往采用逐步放权而非一次性全自动，人工确认和人工接管本身就是可靠性结构的一部分。

回到开头那个"不要过早上多 Agent"的判断——多数场景其实可以先由单 Agent、明确的工具和一张 workflow graph 覆盖，等职责边界稳定之后再引入多 Agent。多 Agent 是一种解法，但很少是第一优先级。

## 5 一个练习 demo：照片工作流 Agent

抽象的原则讲再多，不如一个能把它们全部串起来的练习项目。我推荐的是"照片工作流 Agent"：输入一个照片目录，系统只读扫描文件结构，识别 RAW、JPEG、HIF、XMP 等类型，生成整理和修图计划，并把移动、删除、覆盖 XMP 这类动作标记为需要确认的高风险操作。

| 步骤 | 练习重点 | 示例产物 |
| --- | --- | --- |
| 先写工具，不接 Agent | 工具 schema、路径规范化、只读边界 | `scan_directory`、`summarize_media_files` |
| 接单 Agent | 函数调用、结构化输出、结果压缩 | 目录摘要和行动计划 |
| 加入审批 | 高风险动作识别、pending 状态、dry-run | approval request |
| 加入状态机 | created → scanning → planning → waiting → executing → completed/failed | SQLite 任务表与调用表 |
| 加入 trace | 可回放、可调试、可归档 | `trace.jsonl`、`plan.md` |
| 加入 eval | 越界路径、删除请求、重复执行、工具失败 | 10–30 条回归样本 |

它之所以比一个泛用聊天机器人更有价值，是因为它逼着你把工具、权限、状态、trace、eval 和恢复路径全部走一遍——而这些恰恰就是第 4 节列的那几个真正的难点。

## 6 部署路线

最后是把它跑到云上。部署可以从本地和容器化起步，再逐步迁移到托管容器；越往后，关注点就越从"Agent 行为"转向"secret 管理、对象存储、日志、任务状态和长任务 worker"。

| 阶段 | 架构形态 | 目标 |
| --- | --- | --- |
| 本地 | FastAPI、SQLite、本地 runs 目录、本地 eval | 验证 Agent 行为与 harness 基础 |
| 轻量云端 demo | App Runner、S3、CloudWatch、Secrets Manager、Postgres | 可访问的 Web API 和持久化产物 |
| 准生产 | ECS Fargate（API + worker）、SQS、RDS、S3、OpenTelemetry、Langfuse | 长任务、重试、幂等、观测、审计 |
| 云原生编排 | Step Functions、Bedrock Agents 或自建 workflow runtime | 更复杂的可审计长流程 |

一个常被问到的问题是要不要用 Lambda：它适合承载较小的工具动作，但完整的 Agent runtime 往往需要更长的运行时间、多步状态、审批等待和较完整的 trace，所以早期系统用容器形态通常更顺手。

## 整体而言

如果只留下一句话，那就是：**工业级 Agent 长期积累的对象不是某一句提示词，而是工具、状态、评测、权限、观测和部署运行时共同构成的 harness。** 模型会一代代变强，提示词会反复重写，但你在 harness 上沉淀下来的那套边界、约束和可验证性，才是让一个 Agent 系统从"demo 惊艳"走到"敢交给它干活"的真正分水岭。

构建顺序上，先单 Agent 再多 Agent，先能力再约束，先复用通用机制再自研业务边界；难点排序上，把评测、权限、状态恢复和可观测性放在比"换个更强的模型"更靠前的位置。把这几件事想清楚，剩下的就是耐心地一层层把 harness 打磨出来。
